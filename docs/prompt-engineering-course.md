# Промпты и модели LLM (отчёт по курсу «Промпт-инжиниринг»)

Документ описывает, как в проекте **AI Competitor Analyzer** настроены вызовы больших языковых моделей: **роли сообщений**, **системные и пользовательские промпты**, **ожидаемый формат ответа** и **параметры API**. Реализация: `app/clients/llm_client.py`, сервисы `competitor_filter_service`, `analysis_service`, `report_service`.

---

## 1. Инфраструктура вызова модели

| Параметр | Где задаётся | Значение по умолчанию (пример) |
|----------|----------------|--------------------------------|
| **Модель** | `OPENAI_MODEL` в `.env` / `Settings` | `gpt-4o-mini` |
| **Базовый URL API** | `OPENAI_BASE_URL` | `https://api.openai.com/v1` |
| **Эндпоинт** | код клиента | `POST .../chat/completions` |
| **Temperature** | фиксировано в `LLMClient` | `0.1` (ниже — стабильнее структурированный JSON) |
| **Формат ответа** | `response_format: {"type": "json_object"}` | принуждает ответ одним JSON-объектом (OpenAI-совместимые серверы) |

Сообщение в API всегда из **двух частей**:

- **system** — инструкция роли, правила, схема выхода;
- **user** — контекст задачи: данные поиска, текст страницы, JSON анализов и т.д.

После ответа тело ассистента **парсится как JSON** и валидируется в Python (типы полей, обязательные ключи). Ошибки превращаются в `ExternalServiceError` с коротким текстом для пользователя/API.

## 2. Сводная таблица сценариев (цепочка промптов)

| № | Сценарий | Файл | System prompt (суть) | User prompt (суть) | Выход модели (JSON) |
|---|----------|------|----------------------|--------------------|---------------------|
| 1 | **Фильтрация кандидатов** после Brave Search | `competitor_filter_service.py` | Роль: отобрать релевантные сайты-конкуренты; исключить каталоги, маркетплейсы, соцсети; ответ строго JSON | Ниша, желаемый `site_type`, опционально регион; список кандидатов построчно как JSON с `index`, `title`, `url`, `description` | `{"selected_indices": [int, ...]}` — нулевая нумерация |
| 2 | **Анализ одной страницы** (текст с лендинга) | `analysis_service.py` | Роль: аналитик страницы **только по текстовым полям**; не выдумывать факты; bullets короткие; две шкалы 0–10; выход строго по схеме | Поля `final_url`, `title`, meta, h1, `visible_text` | `positioning`, `offer`, `target_audience`, `strengths[]`, `weaknesses[]`, `design_score`, `animation_potential`, `summary` |
| 3 | **Сводный market summary** по уже сделанным анализам | `report_service.py` | Роль: синтез **уровня рынка** без выдумок; краткие списки сильных/слабых сторон и возможностей дифференциации | JSON-массив структурированных объектов по каждому успешно проанализированному конкуренту | `market_summary`, `common_strengths[]`, `common_weaknesses[]`, `differentiation_opportunities[]` |

Цепочка в продукте: **(1)** опционально после поиска → **(2)** на каждый URL отчёта (после Selenium) → **(3)** один раз по совокупности успешных (2).

## 3. Детали промптов по сценариям

### 3.1. Фильтрация конкурентов (LLM filter)

- **Цель промпта:** сократить шум выдачи Brave до сайтов, похожих на реальных конкурентов в нише.
- **Техники промпт-инжиниринга:**
  - явное **разделение ролей**: что включать / что исключать;
  - **жёсткий контракт вывода**: один объект, ключ `selected_indices`, без markdown;
  - привязка к **типу сайта** из запроса пользователя (`landing`, `personal_brand_site` и др.), чтобы не тащить каталоги, если пользователь искал лендинги.
- **User-часть:** динамически собирается: контекст ниши + построчные JSON по кандидатам + напоминание допустимого диапазона индексов.

### 3.2. Анализ страницы конкурента

- **Цель:** структурированное описание оффера, аудитории, плюсов/минусов и двух **субъективных скоров** только из текста (без vision).
- **Техники:**
  - **ограничение знаний:** «только переданный текст», «не выдумывать метрики и бренды»;
  - **обработка нехватки данных:** просят кратко указать неопределённость;
  - **контроль длины** пунктов strengths/weaknesses (~120 символов);
  - **двуязычие (EN / RU):** два системных промпта (`ANALYSIS_SYSTEM_PROMPT_EN` / `ANALYSIS_SYSTEM_PROMPT_RU`); для интерфейса одиночного анализа без указания языка используется EN; для `reportdemo` — по полю `lang` (по умолчанию RU).
- **User-часть:** шаблон с полями страницы + блок `visible_text` (усечённый на стороне парсера при необходимости).

### 3.3. Сводка по рынку (market summary)

- **Цель:** агрегировать несколько **уже готовых** разборов в один обзор: общие паттерны и идеи отстройки.
- **Техники:**
  - запрет на галлюцинации относительно входных структурированных данных;
  - разделение на абзац (`market_summary`) и три **списка** тем;
  - учет кейса **одного конкурента** (осторожные формулировки);
  - локализация EN/RU аналогично анализу страницы (`MARKET_SUMMARY_SYSTEM_PROMPT_EN` / `_RU`).
- **User-часть:** сериализованный JSON-массив объектов с полями `title`, `positioning`, `offer`, `target_audience`, `strengths`, `weaknesses`, `design_score`, `animation_potential` (без сырого HTML).

## 4. Модель и альтернативы

- В примерах конфигурации по умолчанию указана **`gpt-4o-mini`**: дешевле и быстрее для классификации и JSON-задач при сохранении `json_object`.
- Клиент **OpenAI-compatible**: при смене `OPENAI_BASE_URL` можно направить те же промпты на другой провайдер (при поддержке `chat/completions` и по возможности `response_format`).

---

## 5. Оценка с точки зрения промпт-инжиниринга (кратко)

| Аспект | Что сделано в проекте |
|--------|------------------------|
| Структурированный вывод | Явная схема полей в system; API `json_object`; пост-валидация в коде |
| Снижение галлюцинаций | Правила «только из входа»; temperature 0.1 |
| Многошаговый pipeline | Разные промпты под задачу (фильтр → страница → рынок), без смешивания целей в одном запросе |
| Локализация | Отдельные системные промпты RU/EN для анализа и сводки |
| Отказоустойчивость UI/API | Ошибки парсинга страницы не ломают отчёт целиком; fallback-тексты без stack trace |
